## Bug Description

When two API requests arrive simultaneously (double-click, or multiple browser tabs), both pass the credit check (`checkCredits()`) before either deducts. Both deductions succeed, resulting in the user being charged twice for a single action.

## Root Cause

`checkCredits()` and `deductCredits()` are separate calls with no atomicity guarantee. Between check and deduction, another request can read the same balance.

```
Request A: checkCredits() → balance: 100 → PASS
Request B: checkCredits() → balance: 100 → PASS  (read before A deducted)
Request A: deductCredits(10) → balance: 90
Request B: deductCredits(10) → balance: 80  (should have been rejected)
```

## Fix Required

Replace check-then-deduct with atomic deduction using a PostgreSQL RPC that checks and deducts in a single transaction.

## Acceptance Criteria

- [ ] Credit deduction is atomic (check + deduct in single DB transaction)
- [ ] Concurrent requests: second request fails with "Insufficient credits" if balance is too low
- [ ] Successful deduction returns new balance
- [ ] Failed deduction does NOT modify balance
- [ ] Audit trail: every deduction logged in `credit_transactions` table
- [ ] Rollback: if the action fails AFTER deduction, credits are refunded

## S1: Error States & Validation

| Condition | Behavior |
|-----------|----------|
| Insufficient credits (atomic check) | Return 402 + `{ error: "insufficient_credits", required: X, balance: Y }` |
| Negative deduction amount | Return 400 + `{ error: "invalid_amount" }` |
| User not found | Return 404 |
| Database RPC fails | Return 500 + log error. Do NOT deduct. |
| Rollback after action failure | Refund via `INSERT INTO credit_transactions (amount: +X, reason: 'rollback')` |

## S2: Post-Completion Flow

- **On success**: Deduction logged in `credit_transactions`. Updated balance returned in API response.
- **On rollback**: Refund logged in `credit_transactions` with `reason: 'rollback'`. Balance restored.
- **Idempotency**: Each deduction has a unique `idempotency_key` (request ID). Duplicate requests return the original result, not a second deduction.

## S3: Cross-Feature Integration

- **Content generation**: Calls `deductCredits()` after successful AI response. On AI failure, no deduction.
- **Chat**: Calls `deductCredits()` per message (or per-post if post-creation via function calling).
- **Monthly usage**: `monthly_usage.used_credits` incremented in same transaction as deduction.
- **UI balance display**: Frontend reads balance from API response `creditsRemaining` field. SWR cache invalidated after deduction.

## S4: UX Copy Review

- [ ] Error message: "Not enough credits" not "Insufficient balance" or "402 Payment Required"
- [ ] Rollback notification: "Credits refunded" not "Transaction reversed"

## S6: Manual QA Scenarios

- [ ] Normal deduction: 100 credits → deduct 10 → balance is 90

- [ ] Rollback: Deduct 10 → action fails → credits refunded → balance is original
